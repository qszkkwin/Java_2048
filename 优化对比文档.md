# 2048 小游戏 — 重构优化对比文档

## 一句话概括

> 把一个 562 行的"上帝类"按 MVC 拆分为 7 个职责单一的类，同时修复了绘制性能、算法效率和代码规范三类问题。

---

## 1. 架构：从上帝类到 MVC

**改前** (`demo1/MainFrame.java` 562 行)：

```
MainFrame extends JFrame implements KeyListener, ActionListener {
    数据 (data[][], score)
    逻辑 (moveLeft, checkLeft, checkUp, checkDown...)
    绘制 (paintView: removeAll + new JLabel × 17)
    事件 (keyPressed, actionPerformed)
}
```

一个类同时承担了数据存储、游戏逻辑、界面绘制、事件处理四种职责。

**改后** (`game2048/` 包)：

| 层 | 类 | 职责 | 行数 |
|----|-----|------|------|
| config | `GameConfig` | 魔法数字集中管理 | ~40 |
| model | `GameBoard` | 棋盘数据封装 | ~90 |
| model | `Direction` / `GameState` | 枚举替代 int 标志 | ~10 |
| logic | `GameEngine` | 移动/合并/游戏结束 | ~110 |
| logic | `TileGenerator` + `RandomTileGenerator` | 策略模式生成方块 | ~20 |
| view | `GameBoardPanel` | 自定义双缓冲绘制 | ~120 |
| view | `GameFrame` | 窗口+事件胶水层 | ~90 |



> "原版所有代码堆在一个 JFrame 子类里，数据、逻辑、UI 全耦合。我按 MVC 拆分了——Model 管棋盘数据和得分，View 管绘制和窗口，Logic 作为 Controller 管移动算法和游戏规则。这样每个类只做一件事，单测也能单独写。"

## 2. 绘制性能：从组件式到画布式

**改前**的 `paintView()` 做法：

```java
getContentPane().removeAll();                    // 全部清掉
for (int i...) {
    JLabel tile = new JLabel(new ImageIcon(...)); // 每帧 new 17+ 个对象
    tile.setBounds(...);
    getContentPane().add(tile);
}
```

问题：每按一次方向键 → 创建 17 个 JLabel + 17 个 ImageIcon → GC 压力 → 界面可能闪烁。

**改后**直接覆盖 `paintComponent`：

```java
@Override
protected void paintComponent(Graphics g) {
    super.paintComponent(g);
    Graphics2D g2d = (Graphics2D) g;
    for (int r...) for (int c...) {
        Image img = imageCache.get(path);     // 缓存命中，不重复 IO
        g2d.drawImage(img, x, y, w, h, null); // 零对象创建
    }
}
```

> "Swing 里每 new 一个 JLabel 都在 EDT 上注册一个组件，频繁创建销毁会导致 LayoutManager 反复重算布局，产生闪烁。我改成覆盖 paintComponent，用 Graphics2D.drawImage 直接往画布画，图片用 HashMap 缓存一次加载。本质上是把'组件式渲染'变成了'画布式渲染'，单次按键从创建 17+ 对象降为 0。"

## 3. 算法优化：四个方向的移动如何不写四份

**改前**的做法是为每个方向单独写一套旋转逻辑：

```java
void moveToRight() { horizontalSwap(); moveToleft(); horizontalSwap(); }
void moveToUp()    { Rotate90CounterClockwise(); moveToleft(); RotateArray90(); }
void moveToDown()  { RotateArray90(); moveToleft(); Rotate90CounterClockwise(); }
```

而且 `Rotate90CounterClockwise` 用了 4 步矩阵变换：

```java
void Rotate90CounterClockwise() {
    MatrixTranspose();      // 转置
    horizontalSwap();       // 水平翻转
    reverse2DRowsInPlace(); // 上下翻转
    horizontalSwap();       // 再次水平翻转
}
```

每次都创建新的 `int[4][4]` 数组，然后再拷贝回来。

**改后**用坐标映射公式，一行数学代替四步矩阵操作：

```java
// 所有方向通过旋转统一为"向左移动"
int rotationSteps = switch (direction) {
    case LEFT -> 0; case DOWN -> 1; case RIGHT -> 2; case UP -> 3;
};
// 坐标映射：(row, col) 顺时针旋转 n*90° 后的新坐标
int srcRow = (rotationSteps == 0) ? row : (rotationSteps == 1) ? col : ...;
int srcCol = (rotationSteps == 0) ? col : (rotationSteps == 1) ? size-1-row : ...;
```

在 `mapRow/mapCol` 中用 switch 表达式一次性算出旋转后坐标，直接读写原棋盘，无需创建临时数组。

> "四个方向本质上都是'朝某个方向挤 + 合并'，区别只在于你从哪个角度去看棋盘。我用坐标旋转——把每个方向旋转到等价于向左移动，用数学公式直接算出坐标映射关系，免去了数组拷贝。这样四个方向共享同一套合并逻辑，不会有代码重复。"

## 4. 游戏结束检测：从 4 次数组拷贝到 1 次 O(n²) 遍历

**改前**：

```java
boolean checkLeft() {
    int[][] backup = new int[4][4];    // 拷贝
    copyArray(data, backup);
    moveToleft(CHECK_ONLY);            // 尝试移动
    // 比较...                          // 判断是否变化
    copyArray(backup, data);           // 恢复
}
// checkUp/checkRight/checkDown 各重复一遍
```

每次检查要走 4×(拷贝→移动→比较→恢复) = 16 次棋盘遍历，且创建 4 个临时 `int[4][4]`。

**改后**：

```java
boolean isGameOver() {
    for (int r = 0; r < size; r++)
        for (int c = 0; c < size; c++) {
            if (tiles[r][c] == 0) return false;               // 有空位
            if (c < size-1 && tiles[r][c] == tiles[r][c+1]) return false; // 横向可合并
            if (r < size-1 && tiles[r][c] == tiles[r+1][c]) return false; // 纵向可合并
        }
    return true;
}
```

> "原版游戏结束检测用了'尝试移动法'：对四个方向各拷贝一次棋盘、执行移动、检查变化、再恢复。总共 16 次遍历加 4 个临时 4×4 数组。其实这个问题等价于'有没有相邻相同数字或空位'——一次双重循环 + 三个 if 条件就解决了。O(n²) 从 16n² 变成了 1n²。"

## 5. 代码规范

| 问题 | 改前 | 改后 |
|------|------|------|
| 魔法数字 | `keyCode == 37` `setSize(514, 538)` `setBounds(50+100*i, ...)` | `KeyEvent.VK_LEFT` `GameConfig.WINDOW_WIDTH` `GameConfig.GRID_X + CELL_SIZE*c` |
| 标志位 | `moveToleft(1)` / `moveToleft(2)` 含义不明 | 拆成独立方法，参数语义化 |
| 状态表示 | `int gameState = 1` / `= 2` | `enum GameState { RUNNING, GAME_OVER }` |
| 方法命名 | `RotateArray90` / `moveToleft` / `MatrixTranspose` 混用 Pascal/camelCase | 统一 `lowerCamelCase` |
| 重复注释 | 每个方法 Javadoc 写两遍 | 只写一份，不冗余 |
| 硬编码 | `for (x < 3)` `new int[4]` | `x < len - 1` `new int[arr.length]` |

---

## 总结
> "这个项目我做了四类优化：
> 1. **架构上**，从一个 560 行的上帝类拆成 MVC 三层 7 个类，每个类职责单一；
> 2. **绘制上**，从每帧创建销毁 JLabel 变成了 paintComponent 画布式渲染 + 图片缓存；
> 3. **算法上**，四个方向通过坐标旋转统一为左移，游戏结束检测从拷贝-移动-恢复简化为单次遍历；
> 4. **规范上**，消除了所有魔法数字，用枚举替换 int 标志，统一了命名风格。
>
> 同时保留了原版代码不动作为对比参考，在 `优化对比文档.md` 里有详细的 before/after 对照。"
